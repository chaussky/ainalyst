# Справочник: Трассировка требований (BABOK 5.1)

## Четыре типа отношений — и когда их применять

| Тип | Направление | Смысл | Пример |
|-----|-------------|-------|--------|
| `derives` | сверху вниз | B вытекает из A (иерархия) | BR-001 → SR-003 → FR-007 |
| `depends` | горизонталь | B не имеет смысла без A | FR-007 зависит от FR-005 (авторизация) |
| `satisfies` | требование → компонент | Компонент/модуль реализует требование | FR-007 ← COMP-API-Auth |
| `verifies` | требование → тест | Тест проверяет выполнение требования | FR-007 ← TC-042 |

**Частая ошибка:** помечать всё как `связано` без типа. Тип связи критичен для анализа влияния:
- При CR меняем FR-007 → тип `depends` покажет FR-005 тоже под угрозой
- Тип `verifies` покажет какие тесты нужно перезапустить
- Тип `satisfies` покажет какие компоненты затронуты

## Пресеты формальности

### Lite — Agile, небольшие проекты
Трассируем только:
- `derives`: бизнес-потребность → требование к решению
- Версионность: только major изменения

Не трассируем: `depends`, `satisfies`, `verifies`

Когда выбирать: Agile-проекты до 6 месяцев, нет регуляторных требований, команда < 10 человек.

### Standard — Большинство проектов
Трассируем:
- `derives`: полная вертикаль
- `verifies`: требование → тест
- Версионность: все изменения со статусом

Не трассируем: `depends`, `satisfies` (опционально)

Когда выбирать: Enterprise-проекты, Agile в масштабе, есть QA-процесс.

### Full — Regulated domains, Enterprise
Трассируем все 4 типа отношений.
`rationale` обязателен для каждой связи.
Версионность: полная история изменений.

Когда выбирать: Healthcare, Finance, Government, Safety-critical systems,
проекты с внешним аудитом или compliance-требованиями.

## Признаки проблем — что искать при аудите

### 🔴 Orphan requirement (нет связи вверх)
Требование есть, но непонятно из какой бизнес-потребности оно возникло.
Диагноз: «хотелка», протащенная мимо анализа, или потеряна связь при реструктуризации.
Действие: найти бизнес-потребность или заморозить требование до выяснения.

### 🟡 Нереализованное требование (нет связи вниз)
Требование утверждено, но нет ни компонента, ни теста.
Диагноз: пробел в покрытии или мёртвое требование (не взяли в спринт, забыли).
Действие: уточнить статус у разработчика, добавить в backlog или пометить `deprecated`.

### 🟡 Требование без теста (нет `verifies`)
Реализовано, но не верифицировано.
Диагноз: пробел в QA или тест существует, но не внесён в трассировку.
Действие: проверить у тестировщика, добавить связь или создать тест.

### 🟢 Полное покрытие
Есть: источник (derives ↑), реализация (satisfies/derives ↓), тест (verifies).

## Версионность — паттерн именования

```
FR-001       → текущая актуальная версия (всегда актуальна в requirements)
FR-001_v1.0  → исходная версия (хранится в history)
FR-001_v2.0  → после существенного изменения
```

Правило: при изменении требования — старая версия уходит в `history`, 
в `requirements` остаётся только актуальная запись с новым `version`.

## Типы узлов в репозитории 5.1

Полный список допустимых типов узлов:

| Тип | Описание | Источник |
|-----|----------|----------|
| `business_need` | Бизнес-потребность из 6.1 — **upstream-вершина** для всей трассировки | 6.1 `define_business_needs` |
| `business_goal` | Бизнес-цель из 6.2 — SMART-цель с KPI, второй уровень трассировки | 6.2 `define_goals_and_objectives` |
| `business` | Бизнес-требование — верхнеуровневое требование | 4.3, ручной ввод |
| `stakeholder` | Требование стейкхолдера | 4.3, ручной ввод |
| `solution` | Требование к решению (функциональное/нефункциональное) | 4.3, 7.1 |
| `transition` | Переходное требование (для миграции, запуска) | ручной ввод |
| `test` | Тест-кейс | 7.2, ручной ввод |
| `component` | Архитектурный компонент | 7.4 |
| `user_story` | User Story (Agile) | 7.1 |
| `functional` | Функциональное требование | 7.1 |
| `non_functional` | Нефункциональное требование | 7.1 |
| `business_rule` | Бизнес-правило | 7.1 |
| `use_case` | Use Case | 7.1 |
| `business_process` | Бизнес-процесс | 6.1, 7.1 |
| `data_dictionary` | Элемент словаря данных | 7.1 |
| `erd` | Сущность ERD | 7.1 |
| `change_request` | Запрос на изменение | 4.2 |

**Сквозная цепочка трассировки (от начала до конца):**
```
BN-001 (business_need) → derives → BG-001 (business_goal) → satisfies → FR-001 (solution) → verifies → TC-001 (test)
```
`run_impact_analysis` (5.4) может стартовать от `BN-001` или `BG-001` и показать полный downstream.
`check_coverage` проверяет: все ли BN покрыты хотя бы одной бизнес-целью, все ли BG покрыты требованием.

## Интеграция с другими задачами BABOK

### Входы в 5.1
- **6.1** `define_business_needs` → регистрирует BN-xxx узлы типа `business_need`
  как upstream-вершины. Сквозная трассировка начинается именно отсюда (ADR-054).
- **6.2** `define_goals_and_objectives` → регистрирует BG-xxx узлы типа `business_goal`
  со связями `BG derives BN`. Второй уровень трассировки (ADR-062).
- **4.3** `save_confirmed_elicitation_result` → подтверждённые артефакты выявления.
  Это первичный источник для создания требований в трассировочном репозитории.
- **4.2** `save_cr_elicitation_analysis` → при CR: новые требования или изменения
  существующих. Триггер для обновления трассировки.
- **3.2** Реестр стейкхолдеров → определяет кто является источником требований.

### Выходы из 5.1
- **5.3** Приоритизация использует трассировочный репозиторий: зависимости между
  требованиями влияют на порядок реализации (нельзя приоритизировать B выше A если
  B depends on A).
- **5.4** Оценка изменений вызывает `run_impact_analysis`: граф трассировки — 
  инфраструктура для оценки CR. 5.4 добавляет экспертную оценку поверх технического результата.
- **5.5** Утверждение требований: `export_traceability_matrix` генерирует матрицу
  для формального согласования. Orphan-требования не идут на утверждение.
- **6.x** User Stories, Use Cases, функциональные требования обязательно трассируются
  к бизнес-требованиям. Без трассировки невозможно доказать покрытие.

## Живая трассировка vs мёртвая таблица — как не дать репозиторию протухнуть

Самая частая проблема на практике: матрицу создают в начале проекта и забывают обновлять.

**Правило обновления:** каждое из следующих событий триггерит обновление трассировки:
1. Новое требование добавлено → `init_traceability_repo` или `add_trace_link`
2. Требование изменилось → обновить версию, проверить affected links
3. CR принят → запустить `run_impact_analysis`, обновить затронутые связи
4. Тест создан → добавить `verifies`-связь
5. Компонент изменил scope → проверить `satisfies`-связи

**Кто отвечает за актуальность:** в проекте AInalyst MCP-инструменты
являются «владельцем» репозитория. Каждый вызов `add_trace_link` и обновление
статуса требования автоматически обновляет JSON-файл с датой изменения.
