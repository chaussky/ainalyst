# Справочник: Root Cause Analysis (BABOK 6.1)

## Зачем RCA в контексте 6.1

Самая частая ошибка BA: описывает симптомы вместо корневых причин.

| Симптом | Корневая причина (пример) |
|---------|---------------------------|
| «Жалобы клиентов растут» | Процесс согласования содержит 3 лишних уровня из-за политики 2012 года |
| «Данные теряются при передаче» | Интеграция между CRM и 1С работает через ручную выгрузку Excel |
| «Сотрудники перегружены» | Нет автоматизации приоритизации — всё ставится в очередь вручную |

**Без RCA:** бизнес-потребность формулируется как «нужна система для X»  
**С RCA:** «нужно устранить причину Y, которая приводит к проблеме X»

Второй вариант даёт точный scope решения и позволяет оценить ценность (7.6).

## Три техники RCA

### Техника 1: 5 Почему (Five Whys)

**Когда применять:**
- Проблема линейная (одна цепочка причин)
- Нужно быстро прийти к первопричине
- Команда небольшая, времени мало

**Как применять:**
1. Сформулировать проблему как утверждение (не вопрос)
2. Спросить «Почему это происходит?» → записать ответ
3. Спросить «Почему [ответ 1]?» → записать ответ
4. Повторить до 4–5 итераций (пока ответ не выйдет за рамки наших полномочий менять)
5. Последний ответ — корневая причина

**Пример:**
```
Проблема: Время обработки заявки выросло с 2 до 8 часов

Почему? → Согласование занимает большую часть времени (5.5 ч из 8)
Почему? → Менеджеры ждут своей очереди в общей почте
Почему? → Нет системы уведомлений о заявках на согласование
Почему? → Процесс согласования настраивался вручную под регламент 2012 года
Почему? → Регламент не обновлялся после цифровизации 2019 года

Корневая причина: Регламент согласования устарел, содержит ручные шаги,
несовместимые с текущей инфраструктурой
```

**Ограничения:** не подходит если проблема имеет несколько независимых причин.

### Техника 2: Fishbone / Ishikawa (Диаграмма причин и следствий)

**Когда применять:**
- Проблема комплексная (много возможных причин)
- Нужно систематически охватить все категории
- Есть время на воркшоп со стейкхолдерами

**6 категорий (6M для производства → адаптация для бизнес-анализа):**
| Категория | Описание |
|-----------|----------|
| **People** | Люди, компетенции, мотивация, перегрузка |
| **Process** | Бизнес-процессы, процедуры, рабочие инструкции |
| **Technology** | Системы, инструменты, интеграции, инфраструктура |
| **Policies** | Регламенты, правила, ограничения |
| **Data** | Качество данных, доступность, форматы |
| **Environment** | Внешние факторы: регуляторы, рынок, конкуренты |

**Как применять:**
1. Записать проблему (следствие) в «голове рыбы»
2. Для каждой категории: мозговой штурм — что в этой категории влияет на проблему?
3. Выписать все возможные причины (без оценки)
4. Ранжировать: какие причины наиболее вероятны? (голосование или экспертная оценка)
5. Отобрать 3–5 главных, подтвердить данными

**Пример (частичный):**
```
Проблема: Высокий процент ошибок при передаче данных между системами

Process:  ← Нет формального протокола передачи
          ← Ручная выгрузка Excel как «временное решение» с 2018 года
Technology: ← CRM и 1С не имеют прямой интеграции
            ← Разные форматы дат в системах
People:   ← Разные сотрудники делают выгрузку по-разному (нет стандарта)
```

### Техника 3: Дерево проблем (Problem Tree)

**Когда применять:**
- Нужно связать причины и следствия в единую картину
- Стратегическое изменение с широким охватом
- Нужно показать спонсору «полную картину» проблемы

**Структура:**
- Корни дерева = корневые причины
- Ствол = ключевая проблема (формулировка)
- Ветви/листья = следствия и последствия проблемы

**Как применять:**
1. Сформулировать центральную проблему
2. Вниз: «Почему это происходит?» → корневые причины → ещё глубже
3. Вверх: «Что это вызывает?» → следствия → последствия
4. Проверить: каждая причина действительно ведёт к проблеме? (причинно-следственная связь)
5. Приоритизировать: какие корневые причины устраняемы? какие дают наибольший эффект?

## Нормализованный формат выхода RCA (ADR-056)

Независимо от используемой техники — результат сохраняется в едином формате.
Техника — инструмент мышления. MCP сохраняет нормализованный артефакт.

```json
{
  "rca_id": "RCA-001",
  "problem_statement": "Время обработки заявок клиентов выросло с 2 до 8 часов за последние 6 месяцев",
  "technique_used": "fishbone",
  "root_cause": "Процесс согласования содержит 3 лишних уровня, унаследованных из регламента 2012 года",
  "contributing_factors": [
    "Система согласования не обновлялась с 2012 года",
    "Нет автоматизации уведомлений об ожидающих согласованиях",
    "Менеджеры среднего уровня дублируют проверки руководителей"
  ],
  "evidence": [
    "Среднее время ожидания согласования: 5.5 часа из 8 общих (интервью с командой, 15.03)",
    "78% заявок ожидают согласования более 4 часов (данные системы за Q1 2025)",
    "Регламент согласования датирован 12.04.2012, не обновлялся"
  ],
  "affected_elements": ["capabilities", "policies", "technology"],
  "created": "2025-03-15T10:00:00"
}
```

### Описание полей

| Поле | Описание | Обязательно |
|------|----------|-------------|
| `rca_id` | Уникальный ID: RCA-001, RCA-002, ... | Да |
| `problem_statement` | Чёткая, измеримая формулировка проблемы | Да |
| `technique_used` | fishbone / five_whys / problem_tree | Да |
| `root_cause` | Одна главная корневая причина (не симптом!) | Да |
| `contributing_factors` | Факторы, усиливающие корневую причину | Нет (но рекомендуется) |
| `evidence` | Данные, подтверждающие причинно-следственную связь | Нет (но критично для убеждения спонсора) |
| `affected_elements` | Какие из 8 элементов текущего состояния затронуты | Нет |
| `created` | Дата и время | Да (автоматически) |

### Как отличить корневую причину от симптома

**Тест 1 — Тест «если устраним X, проблема исчезнет?»**
- Если устраним симптом → проблема проявится иначе
- Если устраним корневую причину → проблема уйдёт

**Тест 2 — «Почему X?» можно задать снова**
- Если на ответ можно снова задать «Почему?» → это ещё не корневая причина
- Если «Почему?» больше не имеет смысла → это корневая причина

**Тест 3 — «В наших силах изменить X?»**
- Если причина вне контроля организации → это внешнее воздействие, не корневая причина
- Если можем изменить → это наша корневая причина

## Связь RCA с бизнес-потребностями

После RCA: формулируем бизнес-потребности через инструмент `define_business_needs`.

Схема:
```
RCA-001 (root_cause: "устаревший регламент согласования")
  ↓ связывается с ↓
BN-001 (need: "Оптимизировать процесс согласования заявок")
  ↓ регистрируется в репозитории 5.1 как ↓
Узел business_need типа BN-001
  ↓ становится upstream для ↓
BR-001 (бизнес-требование)
  → FR-001, FR-002 (функциональные требования)
```

Параметр `root_cause_ids` в `define_business_needs` связывает потребность с RCA-артефактами.

## Признаки плохого RCA (чеклист для Claude)

🔴 **Симптом вместо причины:**
- «Нет системы» — это описание решения, не причина
- «Сотрудники не знают как» — симптом, причина — «нет обучения / нет процедуры»

🔴 **Нет доказательств:**
- Корневая причина названа, но без данных — это гипотеза, не факт
- Попросить BA подтвердить: «Откуда мы знаем что это именно так?»

🟡 **Слишком много корневых причин:**
- Если у вас 7+ «корневых причин» — вероятно это contributing factors
- Настоящих корневых причин обычно 1–3

🟡 **Причины вне контроля:**
- «Конкуренты снижают цены» — внешнее воздействие, не корневая причина для RCA
- Такие факторы идут в элемент `external`, не в RCA
